## 2. 功能详情

### 2.1 v1.0.0 - 基础功能（内测版）
**目标**：搭建核心业务流程，验证技术选型。

**完成功能**：
- [x] 基于模拟数据的用户登录/登出
- [x] 上班打卡功能（记录打卡时间）
- [x] 考勤记录的分页查询
- [x] 基础UI/UX设计（Element Plus）
- [x] 前后端联调与基础异常处理

**技术实现**：
- 后端：Spring Boot + MyBatis 基础CRUD
- 前端：Vue 3 + Element Plus 界面，Axios API通信

### 2.2 v1.0.1 - 修复版
**目标**：解决登录认证问题，完善安全性，优化代码质量。

**完成功能**：
- [x] 修复BCrypt密码哈希验证问题
- [x] 修复Element Plus废弃API警告
- [x] 优化用户体验
- [x] 解决Git分支管理问题
- [x] 将硬编码值替换为常量（分页参数、错误码、成功消息等）
- [x] 统一错误码管理，创建ErrorCodeConstants类
- [x] 实现ResponseResult的常量化改造（error和success方法）
- [x] 提升代码可维护性和一致性
- [x] 完善日志记录规范
- [x] 修复API文档与代码不一致问题

**技术实现**：
- 后端：BCrypt加密，统一异常处理，常量管理
- 前端：Element Plus升级，常量统一管理

### 2.3 v1.1.0 - 用户与权限管理（内测版）
**目标**：引入组织架构与权限控制系统，实现多用户管理模式。

**核心功能**：
- [x] 用户注册功能（前端表单验证、后端数据存储）
- [ ] 部门管理（增删改查、层级结构支持）
- [ ] 基于角色的访问控制（RBAC权限模型）
- [ ] 多表数据库设计（用户-部门-角色关联）
- [ ] 管理员面板（用户权限分配、角色管理）
- [ ] 登录认证集成真实用户数据

**技术实现**：
- 后端：Spring Security RBAC，MyBatis 多表关联
- 前端：Vue 3 + TypeScript，路由权限控制

### 2.4 v1.2.0 - 业务规则（内测版）
**目标**：增加考勤规则约束，提升可信度。

**计划功能**：
- [ ] 可配置的打卡规则（上下班时间、迟到/早退阈值）
- [ ] 集成地图API（高德/百度）进行位置验证
- [ ] 规则引擎设计（策略模式）
- [ ] 打卡位置实时验证与记录
- [ ] 合规性检查（如距离公司范围判定）

**技术实现**：
- 后端：规则引擎（策略模式），地图API集成
- 前端：地理位置获取，规则配置界面

### 2.5 v1.3.0 - 数据管理（内测版）
**目标**：满足管理者数据分析和导出需求。

**计划功能**：

**技术实现**：

- 前端：高级表格组件，可视化图表

### 2.6 v1.4.0 - 性能与部署（内测版）

## 3. 技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 后端框架 | Spring Boot 4 | 成熟、社区活跃、易于集成 |
| 前端框架 | Vue 3 + TypeScript | 渐进式、类型安全、组合式API |
| 数据库 | MySQL 8.0 | 稳定、广泛使用，满足业务需求 |
| 权限模型 | RBAC | 灵活、可扩展，适合多角色场景 |
| 缓存 | Redis（v1.4.0引入） | 高性能、支持多种数据结构 |
| 部署 | Docker（v1.4.0引入） | 环境一致性、简化运维 |
| 地图API | 高德/百度（v1.2.0引入） | 国内主流，文档齐全 |

## 4. 依赖与假设

- **外部依赖**：地图API服务需申请密钥，并确保服务可用性。
- **基础设施**：需准备MySQL数据库服务器、Redis服务器（v1.4.0后）。
- **团队能力**：团队成员熟悉Spring Boot、Vue、Git，能快速上手。
- **时间假设**：按每周20小时有效开发时间估算，如有变动需调整计划。

## 5. 开发时间线

### 5.1 迭代周期
每个版本采用 **6周迭代制**：

| 阶段 | 持续时间 | 产出物 |
|------|----------|--------|
| 需求细化 | 0.5周 | 功能清单、技术方案 |
| 开发 | 3周 | 可运行代码、单元测试 |
| 测试 | 1周 | 测试报告、Bug修复 |
| 修复与优化 | 1周 | 稳定版本、性能调优 |
| 文档与发布 | 0.5周 | 版本说明、部署文档 |

## 6. 质量保证

- **单元测试**：核心业务逻辑覆盖率 ≥ 80%（JUnit 5 / Vitest）
- **集成测试**：关键业务流程（登录、打卡、查询）全链路验证
- **代码审查**：每个 PR 至少 1 人审查，确保规范
- **用户验收测试**：每个内测版本邀请 3-5 名真实用户试用
- **性能测试**：v1.4.0 后执行，模拟 1000 并发用户，响应时间 < 2s
- **安全测试**：v2.0.0 前执行，包括 SQL 注入、XSS、CSRF 防护

## 7. 风险评估

| 风险类别 | 风险描述 | 可能性 | 影响 | 应对措施 |
|----------|----------|--------|------|----------|
| **技术** | 地图API集成复杂（坐标系转换、定位精度） | 中 | 高 | 提前调研API文档，准备高德+百度双方案 |
| **资源** | 核心开发人员业余时间不稳定 | 高 | 中 | 每周制定小目标，允许适当缓冲 |
| **技术** | 规则引擎设计超出预期难度 | 中 | 中 | 先实现简单配置，后续迭代优化 |
| **需求** | 用户反馈导致需求变更 | 中 | 低 | 严格控制范围，非核心需求放入后续版本 |
| **协作** | 团队沟通不畅 | 低 | 中 | 建立周会，使用GitHub Projects跟踪 |
| **部署** | Docker环境差异 | 低 | 低 | 多环境测试（开发、测试、生产） |

## 8. 发布标准

每个版本发布前需满足以下条件：
- 所有计划功能已完成并经过测试。
- 单元测试覆盖率不低于80%。
- 无 P0/P1 级 Bug 遗留。
- 文档（README、CHANGELOG、API）已同步更新。
- 通过内部评审。
